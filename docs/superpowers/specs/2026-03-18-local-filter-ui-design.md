# 本地数据三项筛选设计

## 目标

在现有本地数据页面上增加最小改动的筛选能力，不重做布局，不引入复杂筛选面板，只在搜索框下方增加三个可见下拉框和一个确定按钮。

## 交互设计

- 保留现有搜索框，继续用于关键词搜索。
- 在搜索框下方直接显示三个筛选控件：
  - 地区
  - 采购单位
  - 发布时间
- 在三个筛选控件右侧显示一个确定按钮。
- 用户点击确定按钮后，搜索词和筛选条件一起生效并刷新本地列表。
- 不修改当前列表卡片形式。
- 不修改右侧详情区加载和展示方式。
- 筛选只作用于本地数据来源，不扩展到实时查询来源。

## 数据设计

- 复用现有 `GET /api/local/list` 接口。
- 为该接口增加可选查询参数：
  - `region`
  - `project_owner`
  - `publish_range`
- `publish_range` 采用固定枚举值：
  - `all`
  - `today`
  - `7d`
  - `30d`
- 前端初始化加载本地数据时，从返回结果中提取地区和采购单位的去重值，填充下拉选项。
- 发布时间下拉使用固定前端选项，不依赖后端返回选项列表。

## 过滤规则

- 地区：精确匹配 `regionName`
- 采购单位：精确匹配 `projectOwner`
- 发布时间：
  - `today` 过滤为当天发布
  - `7d` 过滤为近 7 天
  - `30d` 过滤为近 30 天
- 搜索词仍按当前模糊匹配逻辑执行，与筛选条件同时生效。

## 错误处理

- 当本地数据目录不存在时，接口继续返回空结果。
- 当字段缺失时，过滤逻辑应安全跳过，不抛异常。
- 当前端筛选条件返回空结果时，沿用现有空状态展示。

## 测试策略

- 为 `GET /api/local/list` 增加接口级测试：
  - 地区过滤
  - 采购单位过滤
  - 发布时间范围过滤
  - 关键词和筛选条件组合过滤
- 前端不新增测试框架，保留为手工验证：
  - 三个下拉框是否渲染
  - 点击确定是否重新请求并刷新列表
  - 切换到实时查询时筛选控件是否禁用或隐藏
